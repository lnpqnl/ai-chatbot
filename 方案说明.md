# AI Chatbot MVP — 方案说明

## 1. 需求理解

题目要求实现一个"可以进行 AI 对话的聊天系统"，核心能力是：

- 用户通过前端页面与 AI 进行流式对话（逐字输出，而非一次性返回）。
- 支持多轮上下文——前面说过的内容在后续对话中仍然有效。
- 支持本地工具调用——AI 能感知可用工具，在合适时机调用，并把结果整合回答。
- 提供 mock 模式，即使没有真实 API key 也能完整跑通流程。

这不是一个"聊天 UI 套一层 API"的问题。真正考察的是：如何设计一条从用户输入到 LLM 响应、再到工具调用回环的完整流水线，并且每一步都是流式的。
【特别地】：我认为，在实际生产中的人力资源很宝贵，最小的MVP不应该是一个一次性的玩具demo，而应该是一个支持可扩展成生产级别的AI-Chat项目，因此，除了业务功能的需求外，还有隐含的可扩展性的需求。

## 2. MVP 范围拆解

我把需求按依赖关系拆成了 6 个阶段，每个阶段都有明确的验证标准（可以用 curl 独立测通）：

| 阶段 | 内容 | 验证方式 |
|------|------|----------|
| 1 | 前后端脚手架 + 健康检查 | `curl /api/health` 返回 OK |
| 2 | SSE 流式对话（mock） | curl 看到逐 token 输出 |
| 3 | 多轮上下文管理 | 连续对话，第二轮提到第一轮内容 |
| 4 | 工具注册 + 调用 + 回环 | 问"几点了"触发 tool_call → tool_result → 文本 |
| 5 | 接入真实 LLM（API） | 替换 provider，功能不退化 |
| 6 | 结构对齐 DDD 分层 | `grep` 验证无跨域 infrastructure 引用 |

每个阶段只做一件事，做完验证（AI验证、人工验证）再进入下一个。这样即使中途断掉，已完成的部分也是完整可用的。避免一次性完成所有功能时导致出现问题时，排查链路过长，不可控的问题。
当然，如果是夜晚时间跑长任务，也可以跳过人工的黑盒验证环节，等最终结果统一验证。

## 3. 第一版做了什么 / 明确不做什么

### 明确做

- **流式 SSE 通信**：POST `/api/chat`，服务端以 SSE 逐 token 推送，前端实时渲染。协议包含 5 种事件类型（token / tool_call / tool_result / error / done），足够表达完整的工具调用流程。
- **Agent 编排循环**：`orchestrate()` 实现 LLM → 判断是否调用工具 → 执行 → 结果回传 → 再调 LLM 的闭环，最多 5 轮迭代，防止无限循环。
- **多轮上下文**：基于 conversation_id 的内存会话存储，首轮不传 id 自动创建，后续通过 id 追加消息。
- **本地工具系统**：`Tool` Protocol + `ToolRegistry` + `TimeTool` 实现，包含参数校验（时区不存在返回错误）和异常捕获。
- **双 Provider**：MockProvider（关键词触发，无需 API key）和 OpenAIProvider（兼容 三方api），通过环境变量切换。
- **DDD 分层**：5 个子域（agent / conversation / llm_gateway / tools / transport），每个子域内3层（application / domain / infrastructure），依赖方向从外到内。
- **前端分层**：1个子域（chat）每个子域内6层types、config → repo → services → runtime（useChat hook）→ UI 组件，App.tsx 只做组合。

【特别地】后端前端的结构分层设计是必须要进行的，AI最开始时急于进入技术方案的设计，这样是不行的，因为这样做出来的只能是一个玩具demo，根本不具备架构维护性、扩展性。持续的迭代，最终大概率会导致项目成为一坨。因此，我协助AI对该需求进行了业务子域的划分：agent编排域、会话域、LLM网关域、工具系统域、消息传输域、日志域（预留）；这样的架构之后，无论是前端还是后端，系统的可维护性、可扩展性均得到了极大的提升。

### 明确不做

- **数据库持久化**：内存存储足够验证 MVP，数据库是基础设施细节，不影响业务逻辑验证。ConversationRepository 已定义 Protocol 接口，切换实现只需替换 infrastructure 层。
- **用户认证**：题目没有要求，且认证是正交关注点，加了只会干扰核心流程演示。
- **多会话管理 UI**：前端只有一个会话页面，没有会话列表/历史记录。MVP 的重点是单会话的完整流程。
- **Markdown 渲染 / 代码高亮**：这是前端展示层的增强，不影响核心通信链路。
- **错误重试 / 断线重连**：SSE 层面只做了基础的 try/catch，没有自动重连机制。
- **并发安全**：内存 repo 没有加锁。单用户场景下不会出问题，多用户需要切到数据库。

## 4. 取舍理由

**SSE vs WebSocket**：SSE 是单向推送，恰好匹配"客户端发一条消息、服务端流式返回"的模式。WebSocket 的双向能力在这个场景下是多余的，而且 SSE 天然兼容 HTTP，不需要额外的连接管理。

**Agent 循环 vs 简单转发**：如果只做文本聊天，一个 `provider.stream()` 就够了。但工具调用要求 LLM → 工具 → LLM 的多轮交互，这必须有一个编排层。`orchestrate()` 就是这个编排层，它把"是否需要工具"的判断权交给 LLM，自己只负责执行和回传。

**Protocol（鸭子类型）vs 抽象基类**：Python 的 Protocol 不要求继承，MockProvider 和 OpenAIProvider 只需实现 `stream()` 方法就满足类型检查。

**内存存储 vs SQLite**：MVP 阶段内存存储的启动速度为零、调试透明、不需要迁移脚本。ConversationRepository Protocol 已经定义好接口边界，后续切换到 SQLite/PostgreSQL 只需要新增一个 infrastructure 实现，不动 domain 和 application 层。

**LangChain vs 直接调 OpenAI SDK**：LangChain 在流式 tool_call 聚合上有成熟的 chunk 处理逻辑（`tool_call_chunks` 的 index 分组），自己实现的话容易遗漏边界情况。但只用了 `ChatOpenAI` + `astream()`，没有引入 Agent/Chain 等重型抽象。

## 5. 核心流程

```
用户输入
  │
  ▼
前端 useChat.handleSend()
  │ POST /api/chat { message, conversation_id? }
  ▼
chat_route: 校验 → 获取/创建会话 → 追加 user message
  │
  ▼
agent_service.run_agent(messages)
  │ 组装 provider + tool_registry
  ▼
orchestrate() 编排循环（最多 5 轮）
  │
  ├── provider.stream(messages, tool_schemas)
  │     │
  │     ├── 输出 str → yield TokenEvent → SSE "token"
  │     └── 输出 ToolCall → 进入工具处理
  │
  ├── yield ToolCallEvent → SSE "tool_call"
  │
  ├── tool.execute(arguments)
  │     ├── 成功 → ToolResult(success=True, data=...)
  │     └── 失败 → ToolResult(success=False, error=...)
  │
  ├── yield ToolResultEvent → SSE "tool_result"
  │
  ├── 将 tool result 追加到 working_messages
  │
  └── 下一轮迭代：带工具结果再调 LLM
        │
        └── LLM 生成最终文本 → TokenEvent → DoneEvent
```

## 7. 多轮上下文处理

- **首轮**：前端不传 `conversation_id`，后端创建新会话（UUID），注入 system prompt，追加 user message。
- **后续**：前端通过 `done` 事件获取 `conversation_id`，后续请求都带上这个 id。后端从内存 repo 加载完整消息历史，追加新 user message 后，把整个 `messages[]` 传给 LLM。
- **LLM 视角**：每次调用都能看到 system + 所有历史 user/assistant/tool 消息，天然支持多轮理解。
- **限制**：没有做 token 截断。当对话过长超过模型上下文窗口时会报错，这是后续优化项。

## 8. 工具调用机制

### 判断

工具 schema 通过 `registry.get_schemas()` 传给 LLM（作为 `tools` 参数）。是否调用工具完全由 LLM 自主决策——模型根据用户意图和 schema 描述判断是否需要调用、调用哪个工具、传什么参数。Mock 模式下则通过正则匹配关键词（时间|几点）模拟这个判断。

### 执行

`orchestrate()` 收到 `ToolCall` 后：
1. 通过 `registry.get_by_name(name)` 查找工具，找不到返回错误。
2. 调用 `tool.execute(arguments)`，整个调用包在 try/except 中。
3. 工具返回 `ToolResult`，包含 success 标志和 data/error。

### 返回

工具结果作为 `role="tool"` 消息追加到 `working_messages`，然后进入下一轮循环。LLM 看到工具结果后生成自然语言回复。前端同时收到 `tool_call` → `tool_result` → `token` 三种事件，可以逐步展示调用过程。

### 参数校验

工具内部负责校验。以 `TimeTool` 为例，传入不存在的时区会返回 `ToolResult(success=False, error="未知时区: ...")`，orchestrate 会将这个错误信息回传给 LLM。

## 9. Mock 模式如何启用

修改 `backend/.env`（没有则需要新建，可以参考backend/.env.example）：

```bash
LLM_PROVIDER=mock #改为mock即可
```

MockProvider 的行为：
- **普通对话**：回显用户消息，多轮时显示"第 N 轮对话"前缀。
- **工具触发**：正则匹配"时间/几点/what time"等关键词，生成 `ToolCall(name="get_current_time")`。
- **工具结果处理**：检测到 `role="tool"` 的最后一条消息后，将内容包装为自然语言回复。

Mock 模式不需要网络、不需要 API key，可以完整验证 SSE → Agent 循环 → 工具调用 → 前端渲染的整条链路。

## 10. 当前限制与后续优化方向

| 限制 | 后续方向 |
|------|----------|
| 内存存储，重启丢失 | 接入 SQLite / PostgreSQL，只需实现新的 Repository |
| 无 token 截断，长对话可能超限 | 引入滑动窗口或摘要压缩策略 |
| 单工具（时间查询） | 注册更多工具（天气、搜索、计算），ToolRegistry 机制已支持 |
| 无认证，所有人共享 | 加 JWT / Session 中间件 |
| 无会话列表 UI | 前端加会话管理页面（新增一个「会话管理」子域接口） |
| 无 Markdown 渲染 | 接入 react-markdown + 代码高亮 |
| 全局 CORS `*` | 生产环境收敛到具体域名 |
| 无自动化测试 | 补充 pytest（后端）+ Vitest（前端）覆盖核心路径 |
| SSE 无断线重连 | 前端加 EventSource 重连 + 后端幂等性保障 |
| 无部署方案 ｜ 适配Docker进行部署即可

## 12. AI 输出的采纳、修改与拒绝

### 采纳的

- **SSE 事件解析代码**（chatRepo.ts）：ReadableStream 逐行解析、buffer 拼接、`data:` 前缀处理。这类模式化代码 AI 写得准确且完整，没有修改直接采纳。
- **LangChain tool_call_chunks 聚合逻辑**（openai_provider.py）：流式 tool call 的 chunk 聚合需要按 index 分组、拼接 args 字符串后 JSON.parse。这个逻辑边界情况较多，AI 的实现覆盖了 id/name/args 分散到达的场景。
- **Pydantic field_validator**（chat_route.py）：请求校验的写法符合 Pydantic v2 规范，直接采纳。
- **BEM 命名的 Less 样式**（chat.less）：变量定义 + 嵌套选择器结构清晰，符合预期。

### 修改的

- **初始的目录结构**：AI 最初生成了扁平结构（所有文件在 domains 根下），我要求改为严格的 application/domain/infrastructure 三层，并在重构阶段逐步对齐。
- **Provider 的 Protocol 定义**：AI 初版用的是 ABC，我改为 Protocol（鸭子类型），更符合 Python 风格且不强制继承。
- **CORS 配置**：AI 生成时没有加 CORS 中间件，在开发联调时前端跨域请求失败，补充了 CORSMiddleware。
- **环境变量命名**：AI 初始用 `ANTHROPIC_*` 变量名（与题目示例不匹配），我统一改为 `OPENAI_*` 以兼容 SiliconFlow 的 OpenAI 兼容接口。

### 拒绝的
- **直接进入技术方案设计阶段**：AI需求确认之后，直接进入了技术细节实现阶段，被我拒绝了，我强制其按照DDD架构设计思想，进行划分了业务子域，确认了架构和项目结构，以满足维护性、可扩展性。
- **引入 LangChain Agent / AgentExecutor**：AI 建议使用 LangChain 的 Agent 框架来管理工具调用循环。我拒绝了，原因是：1）Agent 框架隐藏了编排逻辑，不利于展示理解深度；2）自己写 `orchestrate()` 不到 80 行，控制流完全透明；3）减少不必要的抽象层和依赖。
- **引入 Redis 做会话存储**：AI 建议用 Redis 提高并发性能。MVP 阶段这是过度工程，内存 dict 足够，Protocol 接口已预留扩展点。
- **自动生成 OpenAPI 文档注解**：AI 想给每个端点加 `response_model`、`summary`、`tags` 等 OpenAPI 元数据。这不影响功能，而且会让代码变长，在 MVP 阶段选择不加。
- **前端引入状态管理库（Zustand）**：AI 建议把聊天状态放到全局 store。当前只有一个页面一个会话，`useChat` hook 的 local state 完全够用，引入全局状态管理是过早抽象。
