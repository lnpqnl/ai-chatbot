分层	核心职责	❌绝对禁止的行为❌	可依赖的前置层
UI（React\Vue\nodejs等）	最上层视图层。存放页面、UI 组件、渲染逻辑、用户交互绑定，只做两件事：把状态渲染成视图，把用户操作转发给前置层	直接写数据请求、业务计算逻辑，让前置层反向依赖自己	Runtime
Runtime	运行时 / 状态桥接层。对接状态管理（React Context/Vue Pinia 等）、运行时上下文、生命周期管理，把 Service 的业务逻辑和 UI 层桥接，管理域内运行时状态。	引入 UI 层、写业务逻辑、直接操作数据读写	Types、Config、Repo、Service 层
Service	核心业务逻辑层。存放所有领域规则、数据加工、状态计算、业务流程，是域内的业务核心，只处理逻辑，不碰渲染和状态管理	引入 Runtime/UI 层、硬编码横切功能、把业务逻辑散落到其他层	Types、Config、Repo 层
Repo	仓储 / 数据访问层。封装所有数据读写逻辑：对接后端接口、本地存储、缓存、数据库，对外只暴露数据操作方法，无业务逻辑	引入 Service/UI 等后置层、写业务计算逻辑	Types、Config 层
Config	静态配置层。存放域内环境变量、接口地址、功能开关、业务规则常量、字典配置，无业务逻辑	引入后置层代码、写动态业务逻辑	仅 Types 层
Types	最底层、最稳定的契约层。存放领域模型的 TS 类型 / 接口、枚举、常量、通用类型定义，只有契约，无任何业务实现	引入任何后置层的代码、写实现逻辑	无（不依赖任何其他层）
